📍 项目位置: /Users/weli/works/stack-vs-heap-benchmark

🎯 项目目标:
  基于博客《栈为什么比堆快》，通过 C 语言基准测试、perf 性能分析和
  strace 系统调用追踪，验证栈与堆的性能差异，并结合 Linux 内核源码
  验证理论分析。

📊 核心测试结果（ARM64 + Alpine Linux）:

  性能对比:
    栈分配:  50 ns/次 (基准线)
    堆分配:  70 ns/次 (慢 1.42x，42%)
    堆复用:  48 ns/次 (接近栈，甚至略快)

  系统调用:
    栈: 3 次（仅程序初始化）
    堆: 2332 次（1166 mmap + 1164 munmap）
    差异: 777 倍

  缺页异常:
    栈: 34 次
    堆: 33,190 次
    差异: 975 倍！

  总耗时:
    栈: 3.2 ms
    堆: 73.1 ms
    差异: 23 倍

✅ 验证的博客论点:

  1. 栈分配不触发系统调用 ✓
     - strace 显示运行时零系统调用
     - 只需 sub sp, sp, #N (ARM64)

  2. 堆分配触发 brk/mmap ✓
     - 2332 次系统调用
     - 大块用 mmap，小块用 brk

  3. 缺页路径上栈与堆等价 ✓
     - 单次成本相同，但频率不同
     - 栈页复用，堆每次 mmap

  4. 堆可以接近栈（预分配）✓
     - 堆复用 48ns vs 栈 50ns
     - 差异在分配方式，非访问速度

🎓 关键发现:

  1. 差异主要来自分配方式
     - 栈: 1 条指令 (sub sp)
     - 堆: malloc + 可能的系统调用

  2. 堆的慢不是"访问慢"
     - 预分配堆可接近栈性能
     - 证明差异在分配，非存储介质

  3. 系统调用是主要瓶颈
     - 堆: 2332 次 vs 栈: 0 次
     - 用户态/内核态切换开销

  4. 缺页频率差异巨大
     - 栈页复用（34 次）
     - 堆每次 mmap（33,190 次）

💡 工程启示:

  不是"堆慢，别用堆"，而是:

  1. 理解成本来源
     - 分配 vs 访问
     - 系统调用 vs 用户态
     - 缓存局部性

  2. 选择合适策略
     - 小数据、LIFO → 栈
     - 大数据、动态 → 堆
     - 频繁分配 → 池/Arena

  3. 优化热路径
     - 减少 malloc/free
     - 预分配 + 复用
     - 批量操作
