# 场景3汇编代码分析：验证栈指针在循环中保持不变

## 验证目标

验证场景3中，虽然数组 `char stack_buffer[50 * FIXED_SIZE]` 声明在 `for` 循环内部，但经过编译器优化后，**栈指针 (sp/rsp) 在每次循环迭代时保持不变**，数组空间只分配一次并被重复使用。

## 源代码结构

```c
uint64_t test_large_space_repeated(void) {
    const int REPEATS = 10;  // 重复 10 次

    for (int repeat = 0; repeat < REPEATS; repeat++) {
        // 数组声明在循环内部
        char stack_buffer[50 * FIXED_SIZE];  // 50 * 16KB = 800KB
        volatile char *p = stack_buffer;

        // 访问每个页
        for (int page = 0; page < TOTAL_PAGES; page++) {
            p[page * PAGE_SIZE] = ...;
            p[page * PAGE_SIZE + (PAGE_SIZE - 1)] = ...;
            // ...
        }
    }
}
```

## 汇编代码关键部分

### 1. 函数序言：一次性分配栈空间

```asm
地址          汇编指令                           说明
----------------------------------------------------------------------
10000086c:   stp  d9, d8, [sp, #-0x30]!      保存寄存器，sp -= 0x30
100000870:   stp  x20, x19, [sp, #0x10]      保存寄存器
100000874:   stp  x29, x30, [sp, #0x20]      保存帧指针和返回地址
100000878:   add  x29, sp, #0x20              设置帧指针 (fp = sp + 0x20)

// 栈保护机制检查（调用 __chkstk_darwin 或类似函数）
10000087c:   mov  w9, #0x8030                 w9 = 0xc8030
100000880:   movk w9, #0xc, lsl #16           w9 = 0xc8030 = 819,248 bytes
100000884:   adrp x16, 0x100004000
100000888:   ldr  x16, [x16]
10000088c:   blr  x16                         调用栈检查

// ⭐ 关键：一次性分配整个 800KB 栈空间
100000890:   sub  sp, sp, #0xc8, lsl #12     sp -= 0xc8000 (819,200 bytes)
100000894:   sub  sp, sp, #0x30               sp -= 48 (对齐)
```

**关键发现 1**：
- 地址 `100000890` 的指令 `sub sp, sp, #0xc8, lsl #12` 一次性分配了 **0xc8000 = 819,200 bytes = 800KB** 的栈空间
- 这是整个 `char stack_buffer[50 * 16384]` 数组的大小
- **栈空间只分配一次，不是每次循环迭代都分配**

### 2. 外层循环：重复10次访问

```asm
地址          汇编指令                           说明
----------------------------------------------------------------------
// 外层循环入口（地址 0x100000904）
100000904:   mov  x10, #0x0                   x10 = 页索引 = 0
100000908:   add  x11, sp, #0x28              ⭐ x11 = 数组基地址 = sp + 0x28

// 内层循环：遍历200页
10000090c:   add  w12, w10, #0x42             计算写入值
100000910:   strb w12, [x11]                  写页首
100000914:   strb w12, [x11, #0xfff]          写页尾 (偏移0xfff = 4095)
100000918:   ldrsb w12, [x11]                 读页首
...（读写并累加到全局计数器）...

100000938:   add  x10, x10, #0x1              页索引++
10000093c:   add  x11, x11, #0x1, lsl #12     x11 += 0x1000 (移动到下一页)
100000940:   cmp  x10, #0xc8                  是否访问完200页？
100000944:   b.ne 0x10000090c                 否，继续内层循环

// 外层循环控制
100000948:   add  w8, w8, #0x1                外层计数器++ (w8)
10000094c:   cmp  w8, #0xa                    是否重复了10次？
100000950:   b.ne 0x100000904                 ⭐ 否，跳回外层循环入口
```

**关键发现 2**：
- 地址 `100000908` 的指令 `add x11, sp, #0x28` 将数组基地址设为 **sp + 0x28**
- 由于 sp 在整个外层循环中**从未被修改**，所以 x11 在每次外层循环迭代时都指向**相同的内存地址**
- 地址 `100000950` 的跳转指令跳回 `0x100000904`，**绕过了栈空间分配代码**（0x100000890）

**关键发现 3**：
- 从地址 `100000904` 到 `100000950` 的整个外层循环中，**没有任何指令修改 sp 寄存器**
- 唯一改变的是：
  - `x10`（页索引）在内层循环中从 0 递增到 199
  - `x11`（当前页地址）= sp + 0x28 + x10 * 0x1000
  - `w8`（外层计数器）在外层循环中从 0 递增到 9

### 3. 函数返回：恢复栈指针

----------------------------------------------------------------------
1000009d8:   mov  x0, x19                     准备返回值
1000009dc:   add  sp, sp, #0xc8, lsl #12      ⭐ sp += 0xc8000 (恢复栈空间)
1000009e0:   add  sp, sp, #0x30               sp += 48 (恢复对齐)
1000009e4:   ldp  x29, x30, [sp, #0x20]       恢复帧指针和返回地址
1000009e8:   ldp  x20, x19, [sp, #0x10]       恢复寄存器
1000009ec:   ldp  d9, d8, [sp], #0x30         恢复寄存器，sp += 0x30
1000009f0:   ret                               返回

## 完整的栈指针变化过程

```
初始 sp:  0x1000_0000 (假设)
  |
  v 保存寄存器 (sp -= 0x30)
  |
sp = 0x0fff_ffd0
  |
  v 一次性分配数组 (sp -= 0xc8030)
  |
sp = 0x0ff37_fa0  ← 在整个外层循环（10次迭代）中，sp 保持这个值不变
  |
  | [外层循环第1次]
  |   数组基址 = sp + 0x28 = 0x0ff37_fc8
  |   访问 0x0ff37_fc8 ~ 0x0fffe_fc7 (800KB)
  |
  | [外层循环第2次]
  |   数组基址 = sp + 0x28 = 0x0ff37_fc8  ← 相同地址！
  |   访问 0x0ff37_fc8 ~ 0x0fffe_fc7 (800KB) ← 重复使用相同的栈页
  |
  | ... 重复 10 次 ...
  |
  v 恢复栈空间 (sp += 0xc8030)
  |
sp = 0x0fff_ffd0
  |
  v 恢复寄存器 (sp += 0x30)
  |
sp = 0x1000_0000 (恢复到初始值)
```

## 验证结论

✅ **验证成功**：虽然源代码中数组 `char stack_buffer[50 * FIXED_SIZE]` 声明在 `for` 循环内部，但编译器（gcc -O2）进行了以下优化：

1. **栈空间只分配一次**：在函数序言中一次性分配 800KB（0xc8000 字节）
2. **外层循环不重新分配**：外层循环的跳转指令（b.ne 0x100000904）绕过了栈分配代码
3. **sp 在循环中保持不变**：从进入外层循环到退出，sp 寄存器的值始终是 `初始值 - 0xc8030 - 0x30`
4. **相同内存被重复访问**：10 次外层循环迭代都访问相同的 800KB 栈内存区域

## 性能影响

这个优化使得场景3能够验证**栈空间复用**机制：

- **首次循环**：访问 200 页新栈页 → 触发约 200 次缺页异常
- **后续 9 次循环**：访问相同的 200 页 → 页表已存在，缺页次数 ≈ 0
- **总缺页次数**：约 200 次（而不是 2000 次）

这与场景2形成对比：
- **场景2**：一次性访问 400 页不同的栈页 → 400 次缺页
- **场景3**：重复访问 200 页相同的栈页 → 200 次缺页

## 编译器优化细节

编译器识别出：
1. 数组 `stack_buffer` 的生命周期虽然在循环体内，但其大小是编译时常量
2. 循环之间没有其他栈操作会干扰数组空间
3. 将局部变量提升到函数级别分配，避免重复的栈指针调整

这是一个典型的**循环不变量外提**（Loop-Invariant Code Motion）优化的应用。

## 对比：如果真的每次都重新分配

如果编译器没有优化，汇编代码应该是：

```asm
外层循环:
    sub sp, sp, #0xc8000    // 分配 800KB
    // 访问数组
    add sp, sp, #0xc8000    // 释放 800KB
    // 循环控制
```

但实际的汇编代码是：

```asm
函数序言:
    sub sp, sp, #0xc8000    // 只分配一次

外层循环:
    // 访问数组
    // 循环控制（sp 不变）

函数返回:
    add sp, sp, #0xc8000    // 只恢复一次
```
